What This Is

Paul Shomo proposed the Co-Wiki: a wiki-based warm storage layer for LLM memory where retrieval happens through graph-based spreading activation instead of flat vector search. He published the design as an open invitation to build.

We formalized the math, proved it works, found three corrections, and built production-grade Rust crates from the verified contracts. Then we wired them to a real wiki filesystem backend, added a performance dashboard, and Dockerized the whole thing.

How Spreading Activation Beats Vector Search

Vector search finds documents that are semantically similar to your query. Spreading activation finds documents that are associatively reachable through backlinks. The difference matters when the answer is 2-3 hops away.

Query: "memory sleep consolidation" Vector search finds: memory-consolidation (direct match) Spreading activation finds: memory-consolidation (direct match) spreading-activation (1 hop via backlink) rem-agent (1 hop, links to consolidation) priming (2 hops, through spreading-activation) chunking (2 hops, through memory-consolidation) thinking-fast-and-slow (3 hops, through priming) fault-injection (3 hops, through rem-agent)

The demo wiki has 20 pages across 5 topic clusters (cognitive science, security, distributed systems, AI, reading notes) with organic cross-domain backlinks. Spreading activation traverses these connections in 68 microseconds.

What Was Proven

We formalized 17 claims from the Co-Wiki design, subjected them to adversarial property-based testing (Python hypothesis + Rust proptest), and built a VOPR-style chaos gauntlet that threw 22,500 randomized operations at the invariants.

✓
Linear spreading activation is a contraction mapping (Banach)
✓
Convergence is geometric at rate d
✓
Graph retrieval beats vector retrieval on associative queries
✓
Modified greedy retrieval achieves ≥ ½ of optimal
✓
Variable human-cognitive chunks enable better token efficiency
✓
Human chunk boundaries are more coherent than fixed-token splits
✓
REM decay/prune/dream operators maintain graph health
×
Hard threshold breaks contraction — disproven, sigmoid fix applied
×
Monotonic hop-decay on general graphs — disproven, holds on pure chains only

The two disproven claims were corrections that improved the model. See PROOF.md for the full verification report.

The Demo

The demo UI is a three-panel dashboard built with React. No mocks. Every number comes from the actual Rust backend running spreading activation on a 20-page wiki.

Left Panel: Retrieval

Type a query or click an example pill. The engine ignites initial activation via TF-IDF, propagates through the backlink graph, and selects articles within a token budget using the knapsack algorithm. Results show title, ID, token cost, and link count. Click a result to view it.

Center Panel: Page Viewer

Shows the selected page with its content and clickable backlink pills. Click a backlink to navigate the graph. This is the core wiki experience.

Right Panel: Performance + REM

Performance counters update every 2 seconds: query count, average latency, min/max, mutex lock-wait time in nanoseconds.

Stress test fires 200 queries and shows p50/p95/p99 latency distribution bars.

REM Agent runs a maintenance cycle: decay weakens stale edges, prune removes dormant pages, dream discovers missing backlinks between similar but unconnected articles. The health ring shows graph reachability percentage.

Mutex Indicator

The header shows a kinetic mutex indicator. Pulsing green dot when idle, glowing amber when any operation holds the lock. Lock-wait time is measured in nanoseconds and displayed in the performance counters.

Running Without Docker

If you want to run it natively:

# Terminal 1: Build and start the server
cargo build --release -p cowiki-server
./target/release/cowiki-server demo-wiki

# Terminal 2: Start the UI dev server
cd ui && npm install && npx vite --port 5180

The dev server proxies /api to localhost:3001. Open http://localhost:5180.

Running the Test Suite

# All 133 tests
cargo test

# Just the VOPR chaos gauntlet (41 adversarial tests)
cargo test -p gauntlet

# Wiki backend end-to-end chaos (9 tests, ~40 seconds)
cargo test -p wiki-backend --test vopr

# Python proof suite (37 property tests)
cd proof && pip install -r requirements.txt && python -m pytest tests/ -v
